4. Summary and background of the invention (Address the gap / Novelty)

Summary:
The present invention, designated AEGIS-ZONE, introduces a security paradigm termed "Zero-Trust Physical Enclaves" for computing workstations handling sensitive data. The system couples the lifecycle of cryptographic session keys to two continuously and independently monitored physical conditions: the biological presence of the authorised operator and the electromagnetic integrity of the surrounding airspace. Cryptographic key material is continuously derived from the live biological variance in the operator's peripheral photoplethysmographic (PPG) signal, using a sliding window of raw infrared sensor readings hashed with SHA-3 (Keccak-256) to produce ephemeral 256-bit AES keys. If the biometric signal ceases, the keys are autonomously destroyed within a configurable grace period of 5.0 seconds. Simultaneously, distributed RF scanner nodes monitor the 2.4 GHz spectrum around the workstation; when a non-whitelisted wireless emitter breaches an RSSI threshold of -50 dBm (indicating proximity within approximately 1.5 metres), the system executes an immediate and unconditional key destruction, overriding even an active biometric session. A live webcam feed on the dashboard renders clear video when keys are active and transitions to animated CRT-style static noise when keys are destroyed, providing the operator with an unambiguous visual indication of the security posture. This dual-condition enforcement, where both biological presence and physical airspace integrity must be simultaneously satisfied, represents the core inventive contribution.

Background:
Modern workstation security relies on discrete, event-driven authentication. A user provides credentials at a singular moment, and a session persists until explicit logout or inactivity timeout. This introduces temporal vulnerabilities, as the system cannot verify continuous physical presence or localized airspace integrity. This produces two primary threats: "walk-away attacks" (Physical Absence Exploitation), where an unattended open session is compromised, and Proximity-Based Wireless Attacks ("Evil Twin"), where rogue access points intercept data from nearby workstations. Existing zero-trust network solutions and discrete biometric logins do not address the need for continuous, dual-factor physical verification.

The proposed system addresses these gaps by requiring that cryptographic keys exist only while both the operator's continuous biometric signal and RF perimeter integrity are verified. Violation of either condition triggers immediate, irreversible destruction of session key material.


5. Objective(s) of Invention

The present invention seeks to fulfil six interrelated objectives. The first is to establish continuous, non-intrusive presence verification requiring no repetitive user interaction beyond initial sensor contact. The second is to derive ephemeral, non-reusable symmetric cryptographic keys from the real-time biological variance present in the operator's physiological signal, treated as a continuously changing entropy source rather than a stable biometric template. The third is to implement localised radio frequency perimeter monitoring at workstation granularity, capable of detecting non-whitelisted emitters within a defined physical radius with sub-second response latency. The fourth is to enforce the architectural invariant that the cryptographic session exists only while both the biometric and RF conditions are simultaneously satisfied, with autonomous key destruction upon violation of either condition. The fifth is to provide a unified real-time monitoring interface displaying biometric, RF, and cryptographic status. The sixth is to provide a state-dependent visual output that renders live camera imagery when the session is active and transitions to an opaque interference pattern when keys are destroyed, providing an intuitive, state-dependent visual indication of cryptographic session status.


6. Working principle of the invent (in brief)

The system features a Central Hub (Raspberry Pi 4) supporting distributed sensors and a dashboard. The pipeline includes: Biometric Entropy Acquisition via a PPG sensor; Cryptographic Key Derivation using SHA-3 on a sliding window of PPG readings to produce 256-bit AES keys; Spatial RF Monitoring via 2.4 GHz ESP32 sweeps; and Spatial Threat Evaluation against a -50 dBm whitelist threshold. Autonomous Cryptographic Purge destroys keys if the biometric grace period (5.0 seconds) expires or spatial intrusion occurs. A State-Dependent Visual Output renders clear video during active sessions and static noise upon key destruction.


7. Description of the invention in detail (Include drawing and or photograph as needed)

The system architecture, depicted in Figure 1, comprises three functional layers: the Embedded Hardware Layer, the Central Hub Layer, and the Presentation Dashboard Layer.

7.1 The Embedded Hardware Layer

The Bio-Lock Node consists of an ESP32-WROOM-32 microcontroller interfaced to a MAX30105 Particle and Pulse Oximeter Sensor via I2C (GPIO 21 SDA, GPIO 22 SCL, 400 kHz clock). The infrared LED pulse amplitude is configured to hexadecimal 0x1F for elevated PPG sensitivity, while the red LED is set to 0x0A. The firmware executes a continuous high-frequency polling loop invoking the sensor's infrared reflectance measurement function on each iteration, returning the instantaneous infrared reflectance magnitude that encodes volumetric blood flow changes in the operator's capillary bed. The cardiac R-peak detection algorithm identifies individual heartbeats from the IR waveform, from which the inter-beat interval is computed and the BPM calculated as 60 divided by (delta in milliseconds / 1000.0). A rolling average over the four most recent valid beats filters physiologically implausible values outside the 20-255 BPM range. When the infrared reflectance value falls below 50000, the firmware appends a diagnostic flag "No finger?" to the serial output, signalling sensor absence to the Hub. This flag is crucial to the grace period mechanism: the Hub's serial parser deliberately ignores lines containing this flag, which prevents new readings from entering the sliding window and allows the grace period timer to count down naturally. Each iteration outputs a formatted serial string: "IR=<value> BPM=<bpm> Avg BPM=<avg>".

The two WIDS Scanner Nodes serve as passive electromagnetic surveillance monitors. Upon boot, each node connects to the Hub's dedicated Wi-Fi hotspot ("Aegis_Hub", IP: 192.168.4.1) and establishes a persistent WebSocket connection to the Hub's WIDS ingestion endpoint. Each node executes a complete 2.4 GHz spectrum sweep at 1-second intervals, iterating over all detectable IEEE 802.11 emitters and recording each emitter's BSSID (hardware MAC address), SSID (human-readable network name), and RSSI (received signal strength indicator in dBm). The collected data is serialised into a JSON payload containing the node's unique identifier and an array of detected networks, then transmitted to the Hub over the WebSocket. Both nodes implement automatic reconnection logic: if the Wi-Fi association is lost, the node re-scans and re-associates; if the WebSocket connection drops, the node re-attempts connection with exponential backoff. This dual-node deployment provides spatial diversity, enabling the Hub to fuse RSSI measurements from two different physical locations for more reliable proximity estimation.

7.2 The Central Hub

The Central Hub runs a Python FastAPI web application on a Raspberry Pi 4 Model B (4 GB RAM, Debian GNU/Linux), served by the Uvicorn ASGI server bound to 0.0.0.0:8000. Upon startup, a dedicated background daemon thread opens a serial connection to the Bio-Lock Node at 115200 baud. The hub architecture prioritizes distinct operational threads to isolate cryptographic key management from asynchronous I/O and web server duties, ensuring that delays in data visualization do not obstruct the critical path of the security state evaluators.

7.2.1 Biometric Entropy Ingestion
The serial reader thread continuously reads lines from the Bio-Lock Node via a non-blocking FIFO queue. Lines beginning with "IR=" are parsed to extract the raw infrared reflectance value. Lines containing the "No finger?" diagnostic are deliberately ignored, so that the grace period timer decrements naturally without receiving artificially generated readings. Valid infrared values are appended to a rolling deque with a maximum length of 20 entries. Once at least 10 valid readings have accumulated in the deque, they are passed to the Cryptographic Engine's key derivation function. The raw infrared amplitudes are preferred over the processed Beats Per Minute (BPM) values, as the raw amplitude readings retain the high-frequency microscopic signal noise necessary for true biological entropy generation, preventing artificial heartbeat replay attacks.

7.2.2 The Cryptographic Engine & Key Lifecycles
The Cryptographic Engine concatenates the deque's contents into a comma-separated UTF-8 string, forming a high-entropy seed. It applies a SHA-3 (Keccak-256) cryptographic hash function to this seed, yielding a 32-byte digest that serves as the ephemeral AES-256 symmetric key. This key is used to instantiate an AES-256-CTR stream cipher object, which encrypts and decrypts a protected data payload (e.g., a secured logical volume or an encrypted video feed) in real time. 

The engine maintains a strict monotonically increasing timestamp of the last valid biometric reading. A periodic system-level check function, invoked precisely every 50 milliseconds by a high-priority scheduler, compares the elapsed time against the configurable grace period threshold of 5.0 seconds. If the threshold is exceeded, the key variable in memory is explicitly overwritten with randomly generated zeros (zeroised), the cipher object is deallocated via Python garbage collection enforcement, and all protected memory structures are marked dirty. This immediately renders all protected data cryptographically inaccessible. Crucially, no copy of the key is ever persisted to non-volatile storage; the key exists solely in volatile RAM and is irretrievably lost upon power failure, system restart, or deliberate purge.

7.2.3 Spatial Fusion and Conflict Resolution
The WIDS Engine operates concurrently via the FastAPI WebSocket endpoint. Each connected scanner node transmits its JSON sweep payload, which the engine deserialises and indexes by node identifier. This architecture enables multi-node spatial fusion, acting as a rudimentary physical trilateration system. For each network detected across all nodes, the engine first checks whether the BSSID or SSID appears in the configurable regex-compatible whitelist (default: ["VIT_Secure", "Aegis_Hub", "S-VIT"]). 

For non-whitelisted emitters, the engine computes the maximum RSSI observed across all scanner nodes to avoid false negatives resulting from physical multipath fading, shadowing, or antenna blind spots near a single sensor. If this maximum RSSI across the physical array exceeds the configurable danger threshold of -50 dBm, the emitter is classified as a CRITICAL_BREACH. Upon classification, an inter-process signal is dispatched to the Cryptographic Engine to unconditionally set the AES key to None — independently of and overriding the current biometric authentication state. This cross-domain override is the core architectural invariant: even if the authorised operator is actively present with a valid biometric signal, a rogue wireless emitter penetrating the spatial envelope triggers immediate cryptographic destruction. Stale network entries older than 3 seconds are automatically pruned via a background sweeping thread to prevent phantom detections from triggering lockdowns after an attacker has departed the perimeter.

7.3 The Presentation Dashboard

The dashboard is a Next.js 16 React application that connects to the Hub's WebSocket endpoint and receives real-time system state updates at 20 Hz. The application renders three primary visual components.

The BioLockChart component renders the infrared reflectance waveform on an HTML5 Canvas element. The chart implements auto-ranging amplitude scaling to accommodate the natural variance of PPG signals across different operators. The waveform trace is colour-coded: emerald green during active cryptographic sessions (key derived) and red during lockdown (key is None). This provides an immediate visual heartbeat indicator that doubles as an authentication status display.

The ThreatRadar component displays all wireless emitters currently detected by the WIDS scanner nodes, sorted by received signal strength (strongest first). Each entry shows the SSID, BSSID, RSSI, and the distance estimate computed by the Hub. Whitelisted networks are displayed with a neutral colour. Non-whitelisted networks exceeding the -50 dBm threshold are highlighted in red with a "DANGER" classification label.

The Secure Workspace Feed component integrates a live video stream from the workstation's built-in webcam via the Web MediaDevices API (navigator.mediaDevices.getUserMedia()). The camera stream is acquired at 1280x720 resolution and rendered to an HTML5 video element that occupies the full area of the secure workspace panel. The visual state of this component is governed entirely by the cryptographic session status received from the Hub. When the system status is SECURE, the live camera feed is rendered at full opacity with a pulsing green "LIVE — AES-256 DECRYPTED" status badge overlaid in the lower-left corner. When the status transitions to LOCKED, the live feed's opacity transitions to zero over 500 milliseconds via a CSS transition, simultaneously revealing an overlapping HTML5 Canvas element that renders animated procedural interference. This interference is generated in real-time at 60 frames per second using the Canvas 2D API's createImageData() function, which populates every pixel with a random greyscale value at a semi-transparent alpha of 180/255. Horizontal scanline artefacts are superimposed at 4-pixel intervals, simulating analogue CRT television interference. Over this noise layer, a centred overlay displays a warning icon, the text "FEED ENCRYPTED," and a diagnostic message instructing the operator to restore biometric contact or remove rogue access points. The sudden transition from clear live video to violent static noise is designed to be immediately noticeable and requires no text interpretation.

8. Experimental validation results:

The integrated AEGIS-ZONE prototype was validated in a controlled laboratory environment using a Raspberry Pi 4 Model B (4 GB RAM, Debian GNU/Linux, Python 3.11), three ESP32-WROOM-32 microcontrollers, a MAX30105 PPG sensor, and a smartphone mobile hotspot as the simulated rogue access point. Six structured experiments were conducted, with results summarised below and in Figures 2-5.

Experiment T1 (Bio-Lock Key Derivation Latency): This experiment measured the end-to-end latency from serial reception of a valid IR reading to completion of SHA-3 key derivation. The average latency was approximately 45 milliseconds, comprising approximately 8 ms for serial buffering and parsing and 37 ms for string concatenation, UTF-8 encoding, and SHA-3 (Keccak-256) computation over the 20-element sliding window. This is well within the 100 ms target threshold, confirming that the key re-derivation cycle does not introduce perceptible latency to the user experience. The latency breakdown is visualised in Figure 2(a).

Experiment T2 (Rogue AP Spatial Detection): This experiment characterised the WIDS spatial detection envelope using a smartphone configured as a mobile hotspot with a non-whitelisted SSID. The rogue emitter was moved to various distances from the workstation while the two WIDS scanner nodes recorded RSSI values. During live deployment, non-whitelisted networks were detected at RSSI values of -11 to -19 dBm at close range (within 0.5 metres) and -32 to -44 dBm at medium range (approximately 1.0 metre), correctly triggering CRITICAL_BREACH lockdowns in every case. The system achieved a 100% True Positive Rate (TPR) for all rogue emitters within the 1.5-metre critical radius, with zero missed detections across more than 50 consecutive test cycles.

Experiment T3 (Whitelisted Network Filtering): After configuring the whitelist to include ["VIT_Secure", "Aegis_Hub", "S-VIT"], neither the Hub's own Wi-Fi hotspot (which typically registered RSSI values of -11 to -19 dBm due to physical proximity) nor the ambient campus infrastructure network triggered any false lockdowns, despite consistently registering RSSI values well above the -50 dBm threshold. This confirms a 0% False Positive Rate (FPR) for the whitelisting mechanism. The detection confusion matrix for T2 and T3 is presented in Figure 2(b), showing 7 true positives, 3 true negatives, and zero misclassifications.

 
Figure 2: AEGIS-ZONE Experimental Validation Results — (a) T1 Key Derivation Latency Breakdown, (b) T2/T3 Detection Confusion Matrix, (c) T4/T5 Purge and Recovery Time Distribution, (d) T6 Concurrent CPU Utilisation

Experiment T4 (Grace Period Enforcement): The operator's finger was removed from the MAX30105 sensor at a recorded timestamp. The Bio-Lock Node's serial output immediately transitioned to the "No finger?" diagnostic flag (triggered when IR reflectance falls below 50000). The Hub's serial parser deliberately ignored these lines, preventing new readings from entering the sliding window and allowing the grace period timer to count down from 5.0 seconds. The AES-256 key was purged at approximately 5.03 seconds after the last valid biometric reading, with the 0.03-second variance attributable to the 50 ms polling interval of the periodic check function. This experiment was repeated 20 times, with the purge time consistently falling within the 5.00-5.05 second range. The distribution of purge times across all 20 repetitions is shown in Figure 2(c), demonstrating exceptional consistency in cryptographic termination times without dangerous long-tail outliers.

Experiment T5 (Key Recovery Post-Purge): Following cryptographic purge, the operator restored finger contact with the MAX30105 sensor. The sliding window began accumulating new valid IR readings, and key re-derivation completed in approximately 2.1 seconds, corresponding to the physiological time required for 10 valid cardiac pulse reflections to accumulate at the sensor's optical polling rate, and for the SHA-3 hardware-accelerated algorithm to hash the renewed entropy pool. The statistical boxplot distribution in Figure 2(c) confirms the median recovery time matches this operational requirement, maintaining the flow state for the primary operator without introducing undue delays.

Experiment T6 (Concurrent Throughput on Resource-Constrained Hardware): With all subsystems operating simultaneously in an integrated stress test — USB serial biometric ingestion, dual-node WIDS WebSocket ingestion parsing thousands of beacon frame data points per second, continuous SHA-3 key derivation from the sliding mathematical deque, AES-256-CTR encryption/decryption of an arbitrary data stream, grace period monitoring, and high-frequency 20 Hz dashboard WebSocket streaming — peak CPU utilisation on the Raspberry Pi 4 was constrained to a maximum of 17%, with an OS baseline averaging roughly 4%. Memory consumption remained strictly below 200 MB, proving no memory leaks in the sustained WebSocket persistence channels. The stacked area chart in Figure 2(d) accurately records the per-subsystem CPU structural contribution over a continuous 60-second observation window, validating the system's operational viability on low-power, commercially available Off-The-Shelf (COTS) edge deployment computing hardware, entirely localized to the physical enclave.

9. What aspect(s) of the invention need(s) protection?


Claim 1 (Method — Dual-Condition Cryptographic Session Lifecycle Control): A method for controlling a cryptographic session at a computing workstation, comprising: (a) continuously acquiring a physiological signal from an operator via a biometric sensor associated with the workstation; (b) maintaining a sliding window of the physiological signal; (c) deriving an ephemeral symmetric encryption key by applying a cryptographic hash function to the contents of the sliding window, and repeatedly re-deriving the key as new signal data enters the window; (d) monitoring a radio-frequency perimeter around the workstation using one or more RF scanner nodes; (e) evaluating detected RF emitters against a whitelist and a signal-strength threshold indicative of physical proximity; and (f) destroying the encryption key in system memory when either (i) the physiological signal is absent for longer than a predetermined grace period or (ii) a non-whitelisted RF emitter is detected above the signal-strength threshold; wherein RF-triggered key destruction operates independently of a current biometric authentication state.

Claim 2 (System — Zero-Trust Physical Enclave): A system for securing a computing workstation, comprising: (a) a biometric entropy acquisition subsystem comprising a physiological sensor and a microcontroller, configured to continuously acquire raw biological signal data from an operator; (b) a cryptographic engine configured to derive ephemeral symmetric keys from a sliding window of the biological signal data via a cryptographic hash function, and to destroy the keys when the biological signal is absent for a duration exceeding a predetermined threshold; (c) a spatial RF perimeter monitoring subsystem comprising a plurality of RF scanner nodes positioned around the workstation, each configured to detect wireless emitters and report their identity and signal strength; (d) a threat evaluation engine configured to classify non-whitelisted emitters exceeding a signal-strength threshold as spatial intrusions and to unconditionally destroy the cryptographic session key upon such classification, independently of the biometric state; and (e) a presentation interface displaying the real-time status of both subsystems and the cryptographic session; wherein the system enforces that the encryption key exists only while both a continuous biometric signal and an RF perimeter integrity condition are simultaneously satisfied.

Claim 3 (Biometric Modality): The method of Claim 1, wherein the physiological signal comprises infrared photoplethysmographic reflectance data acquired from the operator's peripheral vasculature, and the sliding window comprises a fixed number of the most recent raw infrared magnitude readings.

Claim 4 (Cryptographic Specifics): The method of Claim 1, wherein the cryptographic hash function is SHA-3 (Keccak-256) producing a 256-bit key, and the ephemeral key configures an AES-256-CTR cipher for real-time encryption and decryption.

Claim 5 (RF Perimeter Specifics): The system of Claim 2, wherein the RF scanner nodes scan the 2.4 GHz IEEE 802.11 band and a signal exceeding the threshold indicates physical proximity within a defined radius.

Claim 6 (Grace Period): The method of Claim 1, wherein the grace period permits transient sensor interruptions without key destruction, and the biometric sensor outputs a diagnostic signal when physical contact is absent, which the system deliberately ignores to allow the timer to expire naturally.

Claim 7 (Cross-Domain Override): The method of Claim 1, wherein RF-triggered key destruction operates as an unconditional override regardless of whether the biometric signal is currently active, such that spatial intrusion and biometric absence are independently sufficient conditions for key destruction.

Claim 8 (Visual State Indicator): The system of Claim 2, further comprising a state-dependent visual output that acquires a live video stream from a camera associated with the workstation, renders it at full opacity when the key is active, and replaces it with a procedurally generated animated interference pattern when the key is destroyed.

Claim 9 (Multi-Node Spatial Fusion): The system of Claim 2, wherein the threat evaluation engine computes a maximum received signal strength across the plurality of scanner nodes for each non-whitelisted emitter, and the classification is based on said maximum exceeding the threshold.

Claim 10 (Alternative Embodiments): The method of Claim 1, wherein the physiological signal is selected from: PPG infrared reflectance, ECG waveforms, camera-based remote PPG, and pulse pressure transduction; the hash function from: SHA-3, SHA-256, and BLAKE3; and the cipher from: AES-256-CTR, AES-256-GCM, and ChaCha20-Poly1305.
