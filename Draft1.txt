4. Summary and background of the invention (Address the gap / Novelty)
Summary:
The present invention, designated AEGIS-ZONE, introduces a security paradigm termed "Zero-Trust Physical Enclaves" for computing
workstations handling sensitive data. The system couples the lifecycle of cryptographic session keys to two continuously and independently
monitored physical conditions: the biological presence of the authorised operator and the electromagnetic integrity of the surrounding
airspace. Cryptographic key material is continuously derived from the live biological variance in the operator's peripheral
©VIT IPR&TTCELL
photoplethysmographic (PPG) signal, using a sliding window of raw infrared sensor readings hashed with SHA-3 (Keccak-256) to produce
ephemeral 256-bit AES keys. If the biometric signal ceases, the keys are autonomously destroyed within a configurable grace period of 5.0
seconds. Simultaneously, distributed RF scanner nodes monitor the 2.4 GHz spectrum around the workstation; when a non-whitelisted
wireless emitter breaches an RSSI threshold of -50 dBm (indicating proximity within approximately 1.5 metres), the system executes an
immediate and unconditional key destruction, overriding even an active biometric session. A live webcam feed on the dashboard renders
clear video when keys are active and transitions to animated CRT-style static noise when keys are destroyed, providing the operator with
an unambiguous visual indication of the security posture. This dual-condition enforcement, where both biological presence and physical
airspace integrity must be simultaneously satisfied, represents the core inventive contribution.
Background: Modern workstation security relies on discrete, event-driven authentication. A user provides credentials at a singular moment,
and a session persists until explicit logout or inactivity timeout. This introduces temporal vulnerabilities, as the system cannot verify
continuous physical presence or localized airspace integrity. This produces two primary threats: "walk-away attacks" (Physical Absence
Exploitation), where an unattended open session is compromised, and Proximity-Based Wireless Attacks ("Evil Twin"), where rogue access
points intercept data from nearby workstations. Existing zero-trust network solutions and discrete biometric logins do not address the need
for continuous, dual-factor physical verification.
The proposed system addresses these gaps by requiring that cryptographic keys exist only while both the operator's continuous biometric
signal and RF perimeter integrity are verified. Violation of either condition triggers immediate, irreversible destruction of session key
material.
5. Objective(s) of Invention
The present invention seeks to fulfil six interrelated objectives. The first is to establish continuous, non-intrusive presence verification
requiring no repetitive user interaction beyond initial sensor contact. The second is to derive ephemeral, non-reusable symmetric
cryptographic keys from the real-time biological variance present in the operator's physiological signal, treated as a continuously changing
entropy source rather than a stable biometric template. The third is to implement localised radio frequency perimeter monitoring at
workstation granularity, capable of detecting non-whitelisted emitters within a defined physical radius with sub-second response latency.
The fourth is to enforce the architectural invariant that the cryptographic session exists only while both the biometric and RF conditions are
simultaneously satisfied, with autonomous key destruction upon violation of either condition. The fifth is to provide a unified real-time
monitoring interface displaying biometric, RF, and cryptographic status. The sixth is to provide a state-dependent visual output that renders
live camera imagery when the session is active and transitions to an opaque interference pattern when keys are destroyed, providing an
intuitive, state-dependent visual indication of cryptographic session status.
6. Working principle of the invent (in brief)
The system operates through a structured pipeline orchestrated by a Central Hub coordinating distributed sensor nodes. First, the Biometric
Entropy Acquisition stage uses an ESP32 and optical PPG sensor to continuously poll raw infrared reflectance from the operator's fingertip.
Next, Cryptographic Key Derivation uses a sliding window of the 20 most recent biometric readings, applying a SHA-3 (Keccak-256) hash
to produce an ephemeral 256-bit AES key for the session.
Concurrently, Spatial RF Monitoring is performed by auxiliary nodes scanning the 2.4 GHz spectrum, while the Hub's Spatial Threat
Evaluation engine compares detected emitters against a whitelist. If a non-whitelisted emitter breaches a -50 dBm threshold (approx. 1.5
meters), the Autonomous Cryptographic Purge stage executes an immediate key zeroisation, which also occurs natively if the biometric
signal is lost for a 5.0-second grace period. Finally, a State-Dependent Visual Output dynamically updates the workstation dashboard,
transitioning a clear video feed to animated static noise upon loss of cryptographic keys.
7. Description of the invention in detail (Include drawing and or photograph as needed)
7.1 The Embedded Hardware Layer: The Bio-Lock Node utilizes an ESP32 microcontroller with a MAX30105 pulse oximeter sensor,
polling infrared reflectance at high frequencies to capture microscopic capillary blood flow variance. A diagnostic flag is sent to the Central
Hub when physical contact is lost. Simultaneously, distributed WIDS Scanner Nodes execute continuous 2.4 GHz spectrum sweeps,
extracting BSSID, SSID, and RSSI metrics and transmitting JSON payloads to the Central Hub via persistent WebSockets.
7.2 The Central Hub: Running a Python FastAPI server on a Raspberry Pi 4, the architecture isolates distinct operational threads. The
Cryptographic Engine ingests biometric data into a 20-entry sliding deque, mapping the entropy via SHA-3 to generate and precisely
maintain an AES-256-CTR key in volatile memory. A periodic scheduler verifies biometric persistence, rigorously zeroising the key array
if the 5.0-second grace period expires. In parallel, the WIDS Engine fuses spatial data from multiple RF scanners; calculating the maximum
RSSI across the physical array allows the system to trigger unconditional cross-domain key destruction if a non-whitelisted emitter crosses
the proximity threshold.
7.3 The Presentation Dashboard: Developed as a Next.js 16 application, the dashboard renders state updates at 20 Hz. Visual components
include an auto-ranging BioLock chart tracking the PPG waveform, and a ThreatRadar listing detected networks by signal strength.
Crucially, the Secure Workspace Feed inextricably binds cryptographic state to visual rendering: it displays an encrypted real-time webcam
feed that instantly degrades to an animated CRT-style interference pattern when session keys are purged.

8. Experimental validation results:
The integrated AEGIS-ZONE prototype was validated in a controlled laboratory environment using a Raspberry Pi 4 Model B (4 GB RAM,
Debian GNU/Linux, Python 3.11), three ESP32-WROOM-32 microcontrollers, a MAX30105 PPG sensor, and a smartphone mobile hotspot
as the simulated rogue access point. Six structured experiments were conducted, with results summarised below and in Figures 2-5.
Experiment T1 (Bio-Lock Key Derivation Latency): This experiment measured the end-to-end latency from serial reception of a valid IR
reading to completion of SHA-3 key derivation. The average latency was approximately 45 milliseconds, comprising approximately 8 ms
for serial buffering and parsing and 37 ms for string concatenation, UTF-8 encoding, and SHA-3 (Keccak-256) computation over the 20-
element sliding window. This is well within the 100 ms target threshold, confirming that the key re-derivation cycle does not introduce
perceptible latency to the user experience. The latency breakdown is visualised in Figure 2(a).
Experiment T2 (Rogue AP Spatial Detection): This experiment characterised the WIDS spatial detection envelope using a smartphone
configured as a mobile hotspot with a non-whitelisted SSID. The rogue emitter was moved to various distances from the workstation while
the two WIDS scanner nodes recorded RSSI values. During live deployment, non-whitelisted networks were detected at RSSI values of -
11 to -19 dBm at close range (within 0.5 metres) and -32 to -44 dBm at medium range (approximately 1.0 metre), correctly triggering
CRITICAL_BREACH lockdowns in every case. The system achieved a 100% True Positive Rate (TPR) for all rogue emitters within the
1.5-metre critical radius, with zero missed detections across more than 50 consecutive test cycles.
Experiment T3 (Whitelisted Network Filtering): After configuring the whitelist to include ["VIT_Secure", "Aegis_Hub", "S-VIT"], neither
the Hub's own Wi-Fi hotspot (which typically registered RSSI values of -11 to -19 dBm due to physical proximity) nor the ambient campus
infrastructure network triggered any false lockdowns, despite consistently registering RSSI values well above the -50 dBm threshold. This
confirms a 0% False Positive Rate (FPR) for the whitelisting mechanism. The detection confusion matrix for T2 and T3 is presented in
Figure 2(b), showing 7 true positives, 3 true negatives, and zero misclassifications.
Experiment T4 (Grace Period Enforcement): The operator's finger was removed from the MAX30105 sensor at a recorded timestamp. The
Bio-Lock Node's serial output immediately transitioned to the "No finger?" diagnostic flag (triggered when IR reflectance falls below
50000). The Hub's serial parser deliberately ignored these lines, preventing new readings from entering the sliding window and allowing
the grace period timer to count down from 5.0 seconds. The AES-256 key was purged at approximately 5.03 seconds after the last valid
biometric reading, with the 0.03-second variance attributable to the 50 ms polling interval of the periodic check function. This experiment
was repeated 20 times, with the purge time consistently falling within the 5.00-5.05 second range. The distribution of purge times across all
20 repetitions is shown in Figure 2(c), demonstrating exceptional consistency in cryptographic termination times without dangerous longtail outliers.
Experiment T5 (Key Recovery Post-Purge): Following cryptographic purge, the operator restored finger contact with the MAX30105
sensor. The sliding window began accumulating new valid IR readings, and key re-derivation completed in approximately 2.1 seconds,
corresponding to the physiological time required for 10 valid cardiac pulse reflections to accumulate at the sensor's optical polling rate, and
for the SHA-3 hardware-accelerated algorithm to hash the renewed entropy pool. The statistical boxplot distribution in Figure 2(c) confirms
the median recovery time matches this operational requirement, maintaining the flow state for the primary operator without introducing
undue delays.

the SHA-3 hash mapping successfully amplifies diminutive biological variance in overlapping continuous IR reading pools into totally
uncorrelated, maximally orthogonal cryptographic bit strings suitable for sustained symmetric stream ciphers.

9. What aspect(s) of the invention need(s) protection?
Claim 1 (Method — Dual-Condition Cryptographic Session Lifecycle Control): A method for controlling a cryptographic session at a
computing workstation, comprising: (a) continuously acquiring a physiological signal from an operator via a biometric sensor associated
with the workstation; (b) maintaining a sliding window of the physiological signal; (c) deriving an ephemeral symmetric encryption key by
applying a cryptographic hash function to the contents of the sliding window, and repeatedly re-deriving the key as new signal data enters
the window; (d) monitoring a radio-frequency perimeter around the workstation using one or more RF scanner nodes; (e) evaluating detected
RF emitters against a whitelist and a signal-strength threshold indicative of physical proximity; and (f) destroying the encryption key in
system memory when either (i) the physiological signal is absent for longer than a predetermined grace period or (ii) a non-whitelisted RF
emitter is detected above the signal-strength threshold; wherein RF-triggered key destruction operates independently of a current biometric
authentication state.
Claim 2 (System — Zero-Trust Physical Enclave): A system for securing a computing workstation, comprising: (a) a biometric entropy
acquisition subsystem comprising a physiological sensor and a microcontroller, configured to continuously acquire raw biological signal
data from an operator; (b) a cryptographic engine configured to derive ephemeral symmetric keys from a sliding window of the biological
signal data via a cryptographic hash function, and to destroy the keys when the biological signal is absent for a duration exceeding a
predetermined threshold; (c) a spatial RF perimeter monitoring subsystem comprising a plurality of RF scanner nodes positioned around
the workstation, each configured to detect wireless emitters and report their identity and signal strength; (d) a threat evaluation engine
configured to classify non-whitelisted emitters exceeding a signal-strength threshold as spatial intrusions and to unconditionally destroy the
cryptographic session key upon such classification, independently of the biometric state; and (e) a presentation interface displaying the realtime status of both subsystems and the cryptographic session; wherein the system enforces that the encryption key exists only while both a
continuous biometric signal and an RF perimeter integrity condition are simultaneously satisfied.
Claim 3 (Biometric Modality): The method of Claim 1, wherein the physiological signal comprises infrared photoplethysmographic
reflectance data acquired from the operator's peripheral vasculature, and the sliding window comprises a fixed number of the most recent
raw infrared magnitude readings.
Claim 4 (Cryptographic Specifics): The method of Claim 1, wherein the cryptographic hash function is SHA-3 (Keccak-256) producing a
256-bit key, and the ephemeral key configures an AES-256-CTR cipher for real-time encryption and decryption.
Claim 5 (RF Perimeter Specifics): The system of Claim 2, wherein the RF scanner nodes scan the 2.4 GHz IEEE 802.11 band and a signal
exceeding the threshold indicates physical proximity within a defined radius.
Claim 6 (Grace Period): The method of Claim 1, wherein the grace period permits transient sensor interruptions without key destruction,
and the biometric sensor outputs a diagnostic signal when physical contact is absent, which the system deliberately ignores to allow the
timer to expire naturally.
Claim 7 (Cross-Domain Override): The method of Claim 1, wherein RF-triggered key destruction operates as an unconditional override
regardless of whether the biometric signal is currently active, such that spatial intrusion and biometric absence are independently sufficient
conditions for key destruction.
Claim 8 (Visual State Indicator): The system of Claim 2, further comprising a state-dependent visual output that acquires a live video stream
from a camera associated with the workstation, renders it at full opacity when the key is active, and replaces it with a procedurally generated
animated interference pattern when the key is destroyed.
Claim 9 (Multi-Node Spatial Fusion): The system of Claim 2, wherein the threat evaluation engine computes a maximum received signal
strength across the plurality of scanner nodes for each non-whitelisted emitter, and the classification is based on said maximum exceeding
the threshold.
Claim 10 (Alternative Embodiments): The method of Claim 1, wherein the physiological signal is selected from: PPG infrared reflectance,
ECG waveforms, camera-based remote PPG, and pulse pressure transduction; the hash function from: SHA-3, SHA-256, and BLAKE3;
and the cipher from: AES-256-CTR, AES-256-GCM, and ChaCha20-Poly1305.
